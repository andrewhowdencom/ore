# Plan: Add `cache_control` and `prompt_cache_key` Hints to the OpenAI Provider

## Objective

Wire cache-affinity hints into `x/provider/openai` so workshop users running against OpenRouter (or Anthropic-via-OpenRouter) get full prefix-cache discounts, and users running against raw OpenAI get prefix-routing affinity via `prompt_cache_key`. Specifically:

1. Add per-invocation options that emit Anthropic-style `cache_control` blocks (OpenRouter / Anthropic-via-OpenRouter) and OpenAI-native `prompt_cache_key`.
2. Read `cached_tokens` (OpenAI native) and `cache_read_input_tokens` / `cache_creation_input_tokens` (Anthropic-via-OpenRouter) from the streaming response and surface them as `artifact.Usage.CacheReadTokens` / `Usage.CacheWriteTokens`.
3. Document host compatibility.

Out of scope: provider-agnostic cache abstractions across non-openai adapters (ore ships only an openai-compatible provider), and Anthropic-specific SDK paths (the block is pass-through via `SetExtraFields`).

## Context

- **File**: `x/provider/openai/openai.go` — The `Invoke` method builds `openai.ChatCompletionNewParams` and currently only sets a handful of fields. The `SetExtraFields` mechanism is already in use for OpenRouter's `reasoning.include` (lines around `p.includeReasoning`); the same mechanism will carry Anthropic-style `cache_control` blocks.
- **File**: `x/provider/openai/openai.go` — Per-invocation options follow a `xxxOption` struct + `WithXxx` constructor pattern that satisfies `provider.InvokeOption` via `IsInvokeOption()`. Examples: `WithTemperature` (line 55), `WithMaxTokens` (line 116), `WithReasoningEffort` (line 85).
- **File**: `artifact/artifact.go` — `Usage` is a cross-cutting core type (lines 198–220) with `PromptTokens`, `CompletionTokens`, `TotalTokens`. It is consumed as a typed struct in `examples/{calculator,filesystem,verifier-chat}/main.go`, decoded generically by `session/serialize.go:118`, and inspected by `x/compaction/compaction.go`. New fields must be additive (zero value safe, JSON-tag-omittable) to keep all consumers building.
- **File**: `x/provider/openai/openai_test.go` — Tests use `mockTransport` + `mockClient` + `mockResponseSSE` + `simpleSSE` / `usageSSE` / `drainArtifacts` helpers. `TestInvoke_ReasoningIncludeField` (line 663) is the closest analogue: it sets up a mock transport, captures the outgoing request body, and asserts the presence/absence of an `extraFields` key. `TestProviderInvoke_UsageChunk` (line 234) is the analogue for response reading.
- **SDK facts** (verified in `github.com/openai/openai-go@v1.12.0`):
  - `ChatCompletionNewParams.PromptCacheKey param.Opt[string]` exists at `chatcompletion.go:2303` with JSON tag `prompt_cache_key`. Setting it is sufficient for OpenAI native prefix routing.
  - `CompletionUsage.PromptTokensDetails.CachedTokens int64` exists at `completion.go:222–226` (the SDK's `CompletionUsage` is reused by `ChatCompletionChunk.Usage`, line 603). The JSON path is `usage.prompt_tokens_details.cached_tokens`.
  - Anthropic-style fields (`cache_creation_input_tokens`, `cache_read_input_tokens`) are not modeled by the SDK and arrive in `CompletionUsage.JSON.ExtraFields`.
- **Project conventions** (from `AGENTS.md`):
  - Provider adapters use `net/http` + `encoding/json` only; no extra SDKs.
  - Functional options pattern for constructors (`Option`) and per-invocation options (`provider.InvokeOption`).
  - Errors wrapped with `fmt.Errorf("...: %w", err)`.
  - Table-driven tests; `go test -race ./...` must pass.
  - Tracing: spans already carry `model` and `thread_id`; new fields can be added to the existing `provider.invoke` span without a new span.

## Architectural Blueprint

Three small, additive changes, plus documentation and tests. The `Provider` struct, the streaming pipeline, and the per-invocation option pattern remain untouched in shape.

1. **Extend `artifact.Usage`** with two new fields: `CacheReadTokens` and `CacheWriteTokens`. Both are `int` with `,omitempty` JSON tags, zero-defaulting when the host does not report them. This is the only change to a core package.

2. **Add two per-invocation options** to `x/provider/openai`:
   - `WithSessionID(string) provider.InvokeOption` — maps to `params.PromptCacheKey`. Useful for OpenAI native prefix routing, and a natural stable identifier for OpenRouter / Anthropic caches even if the host does not use it.
   - `WithCacheControl() provider.InvokeOption` — stamps Anthropic-style `cache_control: {type: "ephemeral"}` blocks on (a) the system message, (b) the last tool definition, and (c) the last user/assistant text content part. Implemented by post-processing the message/tool slices and emitting the result via `params.SetExtraFields`. This mirrors the pi reference implementation (`packages/ai/src/providers/openai-completions.ts:504–549`).

3. **Read cache usage from the streaming response** inside the existing `for stream.Next()` loop. The `chunk.Usage.TotalTokens > 0` branch (the one that currently emits `artifact.Usage`) becomes the read site. Both formats are populated into the same `artifact.Usage` value; consumers get a unified view.

4. **Document host compatibility** in `doc.go` (a one-paragraph table or list) so consumers know which option works against which host.

5. **Tests** covering: (a) `WithSessionID` causes `prompt_cache_key` in the request body; (b) `WithCacheControl` causes the appropriate `cache_control` blocks; (c) `usageSSE` responses with `prompt_tokens_details.cached_tokens` populate `Usage.CacheReadTokens`; (d) `usageSSE` responses with `cache_read_input_tokens` / `cache_creation_input_tokens` (Anthropic-style) populate the corresponding `Usage` fields; (e) the options are no-ops on the request body when unset.

Tree-of-Thought notes on rejected alternatives:

- *Build-time `Option` for SessionID* — rejected. A workshop user with multiple sessions (or a user with parallel threads) needs different cache keys per call. The per-invocation shape also matches `WithTemperature` / `WithMaxTokens` already in the file.
- *Separate `artifact.CacheInfo` artifact* — rejected. Cache hits are a property of usage and live in the same SSE chunk as the totals. Splitting them forces consumers to correlate two artifacts and breaks the existing `TotalTokens > 0` emission site.
- *Configurable `WithCacheControl(messages, tools)` parameters* — rejected for the MVP. The pi reference applies the same three locations unconditionally; making it configurable is a follow-up. The function takes no arguments, matching the issue's "or equivalent" language.

## Requirements

1. `x/provider/openai` exposes a `WithSessionID(string) provider.InvokeOption` that, when supplied, sets `params.PromptCacheKey` on the outgoing request.
2. `x/provider/openai` exposes a `WithCacheControl() provider.InvokeOption` that, when supplied, causes Anthropic-style `cache_control: {type: "ephemeral"}` blocks to appear on the system message, the last tool definition, and the last user/assistant text content.
3. `artifact.Usage` exposes `CacheReadTokens` and `CacheWriteTokens` fields, both `int`, both zero-omitted in JSON.
4. The provider populates `Usage.CacheReadTokens` from `chunk.Usage.PromptTokensDetails.CachedTokens` (OpenAI native) when present.
5. The provider populates `Usage.CacheReadTokens` and `Usage.CacheWriteTokens` from `chunk.Usage.JSON.ExtraFields["cache_read_input_tokens"]` and `["cache_creation_input_tokens"]` (Anthropic-via-OpenRouter) when present.
6. None of the above changes regress the existing test suite (`go test -race ./...`).
7. `x/provider/openai/doc.go` documents which option works on which host (OpenAI native, OpenRouter, Anthropic-via-OpenRouter, other openai-compatible).
8. Unit tests cover: option request shape, no-op behavior when options are not supplied, OpenAI-style usage reading, Anthropic-style usage reading, and the combined case.

## Task Breakdown

### Task 1: Extend `artifact.Usage` with cache read/write fields
- **Goal**: Make the framework's usage artifact report cache hits and writes as first-class fields, additive and backward compatible.
- **Dependencies**: None.
- **Files Affected**: `artifact/artifact.go`, `artifact/artifact_test.go`.
- **New Files**: None.
- **Interfaces**:
  - `type Usage struct { ...; CacheReadTokens int \`json:"cache_read_tokens,omitempty"\`; CacheWriteTokens int \`json:"cache_write_tokens,omitempty"\` }`
  - `MarshalJSON` updated to include both new fields (with the same `output` alias to avoid recursion).
- **Validation**:
  - `go test -race ./artifact/...` passes.
  - `go test -race ./...` passes (downstream consumers of `Usage` must still compile, with new fields simply unprinted in `examples/*/main.go`).
  - JSON round-trip: an `artifact.Usage` with non-zero cache fields round-trips through `json.Marshal` / `json.Unmarshal`; a zero-value `Usage` emits no `cache_*_tokens` keys.
- **Details**:
  1. Add `CacheReadTokens int \`json:"cache_read_tokens,omitempty"\`` and `CacheWriteTokens int \`json:"cache_write_tokens,omitempty"\`` to the `Usage` struct.
  2. Update the anonymous `output` struct inside `MarshalJSON` to mirror the new fields.
  3. In `artifact/artifact_test.go`, extend `TestUsage_MarshalJSON` to assert both new fields appear when non-zero, and are absent when zero. Confirm the existing `Kind` discriminator and the three legacy fields still serialize correctly.
  4. Do **not** change `session/serialize.go`; its `*artifact.Usage` decoder already uses JSON tags and the new fields will round-trip cleanly.

### Task 2: Add `WithSessionID` and `WithCacheControl` InvokeOptions
- **Goal**: Expose per-invocation options that map to `prompt_cache_key` and Anthropic-style `cache_control` blocks.
- **Dependencies**: None.
- **Files Affected**: `x/provider/openai/openai.go`.
- **New Files**: None.
- **Interfaces**:
  - `type sessionIDOption struct{ id string }` with `func (sessionIDOption) IsInvokeOption() {}`.
  - `func WithSessionID(id string) provider.InvokeOption`.
  - `type cacheControlOption struct{}` with `func (cacheControlOption) IsInvokeOption() {}`.
  - `func WithCacheControl() provider.InvokeOption`.
- **Validation**:
  - `go build ./x/provider/openai/...` succeeds.
  - `go test -race ./x/provider/openai/...` passes (existing tests must still pass — the new options are only honored when supplied).
- **Details**:
  1. Place the new option types and constructors adjacent to `WithTemperature` / `WithMaxTokens` / `WithReasoningEffort` so the file's per-invocation option section stays grouped.
  2. In `Invoke`, add two new variables in the option-walk block: `var sessionID string` and `var cacheControl bool`, populated via type assertions against the new option types (mirroring the `temperatureOption` / `maxTokensOption` pattern).
  3. Do **not** mutate the request here. Task 3 wires the variables into the request body; keeping this task to "add the plumbing" makes it hermetic.

### Task 3: Wire cache hints into the outgoing request
- **Goal**: Translate the resolved `sessionID` and `cacheControl` variables into the wire format the hosts expect.
- **Dependencies**: Task 2.
- **Files Affected**: `x/provider/openai/openai.go`.
- **New Files**: None.
- **Interfaces**: None new. Internal behavior: the `openai.ChatCompletionNewParams` value built inside `Invoke` gets two additional post-processing steps.
- **Validation**:
  - `go test -race ./x/provider/openai/...` passes.
  - A focused test (added in Task 6) confirms that supplying `WithSessionID("sess-1")` causes `"prompt_cache_key":"sess-1"` to appear in the captured request body, and that omitting it causes the key to be absent.
  - A focused test confirms that supplying `WithCacheControl()` causes the appropriate `cache_control` blocks in the captured request, and that omitting it leaves the messages and tools untouched.
- **Details**:
  1. **Session ID**: after building `params`, if `sessionID != ""`, set `params.PromptCacheKey = param.NewOpt(sessionID)`. The SDK's existing field is sufficient; no `SetExtraFields` needed for OpenAI native.
  2. **Cache control — system message**: in `serializeMessages`, when `cacheControl` is true, replace the existing `openai.SystemMessage(content)` call with a `ChatCompletionSystemMessageParam{ Content: openai.ChatCompletionSystemMessageParamContentUnion{ OfContentParts: []openai.ChatCompletionContentPartTextParam{ { Text: content, CacheControl: openai.CacheControlEphemeral } } } }` (or whatever the SDK's spelled-out form is — verify against `chatcompletion.go`; the reference is `openai.CacheControlEphemeral` and `CacheControl: openai.CacheControlEphemeral` on the part). The fallback when `cacheControl` is false is the existing `openai.SystemMessage(content)` call.
  3. **Cache control — last tool**: in `serializeTools`, when `cacheControl` is true and the index is the last element, set `toolParams[i].Function.CacheControl = openai.CacheControlEphemeral`. When false, leave the field at its zero value (the SDK's `omitzero` tag will exclude it from JSON).
  4. **Cache control — last user/assistant text content**: in the `state.RoleUser` and `state.RoleAssistant` (text-only) branches of `serializeMessages`, when `cacheControl` is true, emit the message as a content-parts array with a `CacheControlEphemeral` stamp on the (single) text part. When false, keep the string form. Threading the `cacheControl` flag into `serializeMessages` / `serializeTools` is the cleanest approach: extend the method signatures with a trailing `cacheControl bool` parameter (zero value preserves existing behavior for any test that calls them — but they're unexported, so the only caller is `Invoke`, and updating the single call site is enough).
  5. **No-op verification**: confirm that, with `cacheControl == false`, the JSON shape of the request is byte-identical to today for the same input. This is the regression guard for hosts that ignore unknown fields.
  6. The `OpenRouter` heuristic and `reasoning.include` injection are unchanged.

### Task 4: Read cache usage from the streaming response
- **Goal**: Populate `Usage.CacheReadTokens` and `Usage.CacheWriteTokens` from both OpenAI native and Anthropic-style fields on the streaming response.
- **Dependencies**: Task 1.
- **Files Affected**: `x/provider/openai/openai.go`.
- **New Files**: None.
- **Interfaces**: None new. Internal behavior: the `if chunk.Usage.TotalTokens > 0` branch in `Invoke` reads two extra fields before emitting `artifact.Usage`.
- **Validation**:
  - `go test -race ./x/provider/openai/...` passes.
  - A focused test (Task 6) confirms that an SSE chunk whose `usage` includes `prompt_tokens_details: { cached_tokens: 42 }` causes `Usage.CacheReadTokens == 42`.
  - A focused test confirms that an SSE chunk whose `usage` includes `cache_read_input_tokens: 100, cache_creation_input_tokens: 250` causes `Usage.CacheReadTokens == 100` and `Usage.CacheWriteTokens == 250`.
  - A focused test confirms that an SSE chunk with neither populates the new fields as zero.
- **Details**:
  1. Inside the `if len(chunk.Choices) == 0` block, after reading `chunk.Usage.PromptTokens`, `CompletionTokens`, `TotalTokens` into local variables, also read:
     - `cacheRead := chunk.Usage.PromptTokensDetails.CachedTokens` (OpenAI native; `int64`).
     - For Anthropic-style: parse `chunk.Usage.JSON.ExtraFields["cache_read_input_tokens"]` and `["cache_creation_input_tokens"]` via `json.Unmarshal([]byte(field.Raw()), &n)` (mirror the existing `reasoning_content` parsing idiom a few lines below).
  2. Prefer the Anthropic-style fields when present, fall back to OpenAI native — but in practice, only one of the two hosts will ever set the values, so an `if/else` over the two paths is fine. Concretely: if `cache_read_input_tokens` is present, use it (and its sibling for write); otherwise use `PromptTokensDetails.CachedTokens`. This gives Anthropic-via-OpenRouter precedence on hosts that surface both.
  3. Emit `artifact.Usage{ PromptTokens: ..., CompletionTokens: ..., TotalTokens: ..., CacheReadTokens: int(cacheRead), CacheWriteTokens: int(cacheWrite) }` on the channel.
  4. No new tracing events. The `provider.invoke` span already records the request; usage is application-visible, not trace-visible.

### Task 5: Document host compatibility in `doc.go`
- **Goal**: Make it obvious to a composer which option works against which host, fulfilling the issue's acceptance criterion #4.
- **Dependencies**: None (documentation only). Logically follows Task 2/3/4 so the documented behavior matches what was actually implemented.
- **Files Affected**: `x/provider/openai/doc.go`.
- **New Files**: None.
- **Interfaces**: None.
- **Validation**:
  - `go vet ./x/provider/openai/...` is clean.
  - The package doc renders in `go doc github.com/andrewhowdencom/ore/x/provider/openai` with the new section.
- **Details**:
  1. Add a "Cache control" section to the package doc that describes, in a small table or list:
     - **OpenAI native**: `WithSessionID` is honored (maps to `prompt_cache_key`); `WithCacheControl` is a no-op (the host ignores `cache_control`).
     - **OpenRouter**: `WithSessionID` is honored; `WithCacheControl` is honored and produces full Anthropic-style cache discounts.
     - **Anthropic-via-OpenRouter**: same as OpenRouter.
     - **Other openai-compatible hosts**: best-effort. Unknown fields are ignored; no regression.
  2. State that `Usage.CacheReadTokens` and `Usage.CacheWriteTokens` are populated from the response when the host reports them, and remain zero otherwise.
  3. Keep the section concise; the implementation details live in `openai.go`.

Verify the JSON paths against the SDK's `CompletionUsage` and `CompletionUsagePromptTokensDetails` shapes (e.g., `usage.prompt_tokens_details.cached_tokens` and top-level `usage.cache_read_input_tokens` / `usage.cache_creation_input_tokens`).
  2. **`TestInvoke_WithSessionID`** — table-driven, two cases (option supplied, option not supplied). Uses `recordingMockTransport` (already in the test file) to capture the outgoing body. Asserts the presence/absence of `prompt_cache_key`.
  3. **`TestInvoke_WithCacheControl`** — table-driven. Two cases (option supplied, option not supplied). Captures the body and asserts:

## Risks & Mitigations

| Risk | Impact | Low/Medium/High |
|---|---|---|
| The openai-go SDK does not yet expose `CacheControlEphemeral` as a typed constant (or the JSON shape for content parts has changed) | High | Low | Verify against `chatcompletion.go` in the pinned SDK version before writing Task 3. If the field is not yet present, fall back to `params.SetExtraFields` for the entire message/tool slice (the same path used for `reasoning.include`), with the cost of losing the SDK's typed surface. |
| Anthropic-via-OpenRouter's `cache_read_input_tokens` arrives as a string, not an integer, in some hosts | Medium | Low | Use `json.Number` (or `int64` with the SDK's `param.Opt` machinery) and tolerate both. Add a defensive test in Task 6 covering both wire shapes. |
| Adding fields to `artifact.Usage` breaks a downstream consumer that does exhaustive struct matching (e.g. a `switch u := u.(type)` over a closed set) | Medium | Low | `Usage` is a struct, not an interface; no exhaustive matching is possible. JSON tags use `omitempty` so serialized payloads are stable for the existing three fields. The `session/serialize.go` decoder uses the JSON tags, so the new fields are decoded transparently. |
| `WithCacheControl` mutates the request shape in a way that breaks an openai-compatible host that *partially* understands the field (e.g. parses `cache_control` but rejects `tools[i].function.cache_control`) | Medium | Low | The `SetExtraFields` escape hatch is a pass-through; the host decides. The no-op test (Task 6 case 3) confirms the request shape is byte-identical when the option is not supplied. Document the risk in `doc.go` under "Other openai-compatible hosts". |
| `provider.Invoke`'s `chunk.Usage` is the streaming accumulator's last frame, not a per-frame total, so cache fields are only present on the final usage chunk | Low | High | The existing code already gates emission on `TotalTokens > 0`, which is true only on the final chunk. The same gate works for cache fields. No change to the streaming strategy is required. |
| The host reports both `PromptTokensDetails.CachedTokens` and `cache_read_input_tokens` (some OpenRouter models surface both) | Low | Medium | Task 4 specifies Anthropic-via-OpenRouter precedence: if `cache_read_input_tokens` is present, use it; otherwise fall back to OpenAI native. This is consistent with the rest of the field naming (Anthropic fields win on hosts that speak Anthropic). |
| Existing example binaries (`examples/calculator`, `examples/filesystem`, `examples/verifier-chat`) print `Usage` fields and silently drop the new ones | Low | High | Acceptable. Their print statements are illustrative, not contractual. A follow-up could extend them, but it is **not** in scope for this plan per the issue's "Scope" section. The plan does, however, verify the examples still build via `go test -race ./...`. |
