# Plan: feat(x/slash): ephemeral feedback, raw parsing, and /help

## Objective

Close the gaps in the `x/slash` command framework by implementing ephemeral feedback, raw parsing, and auto `/help`.

## Background

The `x/slash` command framework supports three behavioral patterns but has gaps:

1. **Parsing**: `Intercept` splits input with `strings.Fields`, which mangles commands like `/include /path/with spaces`.
2. **Ephemeral feedback**: Commands like `/help`, `/status`, or unknown-command fallback need to show a message to the user without triggering inference and without adding a turn to the conversation history.

## Design

### 1. New handler signature

```go
type Command struct {
    Name  string   // e.g. "help" (no leading slash)
    Input string   // everything after the command, raw and unmodified
}

type Result struct {
    Replace  session.Event   // nil = consume, non-nil = continue with this event
    Feedback artifact.Text   // single ephemeral UI message, not persisted
}

type Handler func(ctx context.Context, cmd Command) (Result, error)
```

`Input` preserves whitespace. `Fields` is a convenience helper; callers can bring their own parser.

### 2. Feedback path

Introduce `loop.FeedbackEvent` that flows through the `FanOut` to conduits but is **not appended to state** by `EventBus.Emit`. Conduits render it as a `System`-styled message — scrollable, but invisible to the LLM on the next turn.

### 3. Auto `/help` and unknown-command fallback

- `Bind` gains a `description` parameter so every command can self-document.
- `NewRegistry()` auto-registers `/help` that lists all bound commands and their descriptions.
- Unknown commands show a feedback message: `Unknown command: /foo. Type /help for available commands.`

### 4. Break `session.Interceptor` to match

```go
type InterceptResult struct {
    Event    Event
    Feedback []artifact.Text
}

type Interceptor interface {
    Intercept(ctx context.Context, event Event) (InterceptResult, error)
}
```

`x/slash` converts `slash.Result` → `session.InterceptResult` in its `Intercept` implementation. `Stream.processOne` checks `result.Event == nil` to decide whether to consume or continue.

## Tasks

### Task 1: Update `session.Interceptor` interface

- Change `session/event.go` to replace `Interceptor` interface returning `(Event, bool, error)` with returning `(InterceptResult, error)`.
- `InterceptResult` contains `Event` and `Feedback []artifact.Text`.
- Update `InterceptorFunc` to match.

### Task 2: Add `loop.FeedbackEvent`

- Add `FeedbackEvent` struct to `loop/loop.go` with `Content string` and `Ctx context.Context`.
- Implement `Kind() string` returning `"feedback"`.
- Implement `Context() context.Context`.
- Implement `MarshalJSON` for JSON serialization.
- Ensure `EventBus.Emit` does NOT append `FeedbackEvent` to state (it doesn't append anything except `TurnCompleteEvent`, so no change needed there).

### Task 3: Update `x/slash` handler signature and parsing

- Change `Handler` signature to `func(ctx context.Context, cmd Command) (Result, error)`.
- Add `Command` struct with `Name` and `Input` fields.
- Add `Result` struct with `Replace` and `Feedback` fields.
- Add `Fields(input string) []string` convenience helper (using `strings.Fields` for backward compatibility, but callers can bring their own parser).
- Update `Intercept` to use raw `Input` instead of `strings.Fields` parsing.
- Update `Intercept` to return `session.InterceptResult` instead of `(Event, bool, error)`.
- Convert `slash.Result` to `session.InterceptResult` in `Intercept`.

### Task 4: Wire feedback into `session.Stream.processOne`

- Update `Stream.processOne` in `session/stream.go` to handle the new `Interceptor` interface.
- Check `result.Event == nil` to decide consumption (vs old `consumed bool`).
- If `result.Feedback` is non-empty, emit `loop.FeedbackEvent` for each feedback item via `s.step.Emit`.

### Task 5: Auto-register `/help` and unknown-command fallback

- Update `slash.Bind` to accept a `description` parameter.
- Auto-register `/help` in `NewRegistry()` that lists all bound commands and their descriptions.
- Implement unknown-command fallback that returns a feedback message: `Unknown command: /foo. Type /help for available commands.`

### Task 6: Update TUI to render feedback events

- Add `feedbackMsg` Bubble Tea message type in `x/conduit/tui/model.go`.
- Handle `loop.FeedbackEvent` in the TUI's event subscription goroutine in `x/conduit/tui/tui.go`.
- Send `feedbackMsg` to the program when feedback events arrive.
- Handle `feedbackMsg` in `model.Update` to render as a system-styled message in the conversation.
- Add "feedback" to the TUI's subscription kinds.

### Task 7: Update HTTP conduit to serialize feedback events

- Add `feedbackEventJSON` struct in `x/conduit/http/types.go`.
- Update `MarshalOutputEvent` to handle `loop.FeedbackEvent`.
- Update `UnmarshalOutputEvent` to handle `loop.FeedbackEvent`.
- Add "feedback" to default event kinds in `sendMessage` and `sessionEvents` handlers.

### Task 8: Update existing callers and tests

- Update `examples/http-chat/main.go` to use the new `slash.Bind` signature (add description parameter).
- Update `x/slash/slash_test.go` to use new handler signature and test feedback/help/unknown commands.
- Update `session/stream_test.go` to use new `Interceptor` interface.
- Update `session/event_test.go` if needed for `InterceptResult`.
- Update `session/integration_test.go` if needed.

## Validation Criteria

- All tests pass: `go test -race ./...`
- The slash command parsing preserves whitespace in the input (e.g., `/include /path/with spaces`)
- Feedback events are not persisted to state but are visible to conduits
- `/help` command is auto-registered and lists all bound commands
- Unknown commands show a feedback message without triggering inference
- Both TUI and HTTP conduits render/serialize feedback events
